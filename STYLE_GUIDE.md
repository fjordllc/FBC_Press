# FBC Press Style Guide

このガイドは「やってはいけないこと」のリストではありません。  
**FBC Press の章が、なぜ読む価値を持つのか**を定義し、その水準を再現するための基準です。

チェックリスト(`REVIEW_CHECKLIST.md`)は最後の点検にだけ使ってください。**チェックリストを全部通っても、退屈な章はいくらでも書けます。** 先に読むべきはこの章の「第0節」です。

---

## 0. まず読む — 良い章と悪い章の違い

### 0.1 この水準を目指す

FBC Press の到達点は、既刊『Tailwind CSS 教科書』第1章です。1 章で次をすべてやっています。

- 見出しが **主張そのもの**(例:「CSS が大規模化で壊れる理由」「命名規則による戦い — BEM の登場と限界」)
- コード例が **積み上がる**。`<font>` → 関心の分離 → BEM → CSS-in-JS → ユーティリティ、と前の例を踏まえて次が出る
- **歴史と対比表** が議論の中で必然性を持って現れる
- 「これは本当に分離と呼べるのか?」と **読者と一緒に疑う**
- 章末で次章へ、本文中で他章へ、**体系として接続**する

これを「密度のゴールドスタンダード」と呼びます。新しい章を書く前に必ず読んでください。

### 0.2 これが「ダメな章」だ — 実際の失敗例

過去に、このガイドだけを与えて書かせた本(『HTML 雑学』)は、全章が次のようになりました。**同じ轍を踏まないために、何が起きたかを正確に知ってください。**

悪い章の「まとめ」は、**全章でタイトルを差し替えただけの同一文**でした。

```text
この章で重要なのは、〇〇〇 を単なる豆知識ではなく、HTML の歴史・仕様・
実装の文脈で捉えることです。細かな書き方や短い標語だけを覚えるのでは
なく、どの条件でそう言えるのかまで含めて理解すると…
```

見出しも全章で同一でした。

```text
## 5.1 まず押さえたいこと
## 5.2 具体例で見る
## 5.3 なぜそうなるのか
## 5.4 まとめ
```

各章は 25〜30 行。コード例は自明な 1 スニペットだけ。これは「章」ではなく **テンプレートの穴埋め** です。授業を聞いた人のノートを覗いているような、単純でつまらないまとめになります。

### 0.3 なぜ穴埋めは起きるのか — 書き手が陥る罠

このガイドや章テンプレートを **記入フォーム** として読むと、必ず穴埋めになります。「節を6つ作って、それぞれ"なぜ"を1個入れて、参考資料を付ける」——条件を満たすことだけを目標にすると、**最小限で満たして止まる**からです。

順序を逆にしてください。**先に「この章で読者に何を分からせ、何を判断できるようにするか」を1文で言い切る。** その主張を成立させるために必要な議論を組み立てる。テンプレートは、組み上がった議論を **後から整理する道具** であって、議論を生む道具ではありません。

### 0.4 密度の下限(これだけは守る)

「短く」「最小限」は **1文と1コードブロックの話** です。**章全体が薄くてよい、という意味ではありません。** 次を満たさない章は、まだ書き終わっていません。

- 章の主張を、**具体例・対比・根拠** の3点セットで支えているか。主張の言い換えだけで段落を埋めていないか
- コード例や事実が、前のものを踏まえて **積み上がっているか**。並列に置いただけになっていないか
- 読者が「なるほど、だからこうなのか」と **一段深く理解する瞬間** が、章のどこかにあるか
- その章にしか書けない **固有の中身** があるか。タイトルを別の語に変えても通る一般論になっていないか

## 1. 文体ルール

- 基本は `です・ます調`。断定はするが、語気は強すぎない。
- 1 文は短めに保つ。1 段落 1 論点が原則。**ただし章全体の密度はこれと無関係。短い文を、十分な数だけ積む。**
- 抽象論だけで進めず、すぐ短い具体例を置く。
- 説明は「何か」だけで終えず、必ず「なぜそうなるか」「なぜそれが問題か」まで言う。
- 読者を置いていかない。初出用語は括弧で短く言い換える。
- 読者への語りかけは控えめだが明確。ただし **同じ定型句(「覚えておきましょう」等)を全章で機械的に繰り返さない。** その場の文脈に合った言葉を選ぶ。
- 煽らない。流行語や過度な比喩は少ない。
- 内容の善悪を単純化しない。「絶対にダメ」より「なぜ痛むか」「いつなら許容できるか」を示す。
- 既存の考え方や旧バージョンを扱うときは、誤差なく条件を限定する。
- 強調は多用しない。本当に重要な一文だけ太字化する。

## 2. 構成ルール

- 本全体は `部 → 章 → 節` の三層構造。
- 各部には導入ページを置き、その部の目的と読み方を先に示す。
- 各章は `章タイトル → 導入 → 議論の展開 → 参考資料` の流れ。**節の見出しと数は章ごとに変える。固定の雛形を全章に当てはめない。**
- 節番号は必ず振る。`1.1` のように読者が参照しやすい粒度にする。
- **節見出しは「役割名」ではなく「主張・論点」にする。** `5.1 まず押さえたいこと` は禁止。`5.1 表は行を並べるだけの構造ではない` のように、見出しだけで何を言う節か分かるようにする。
- 章末には必ず `## 参考資料` を置き、一次情報を優先して列挙する。
- 部末には演習を置き、暗記確認ではなく説明・比較・判断を問う。
- 本編で詰め込みすぎるときは、付録へ逃がす。導入手順、用語集、チートシート、実践ミニ例は付録に向く。
- 章の中では、概念説明のあとに実務判断やアンチパターンへつなぐ構成が強い。

## 3. 説明の順序

- 基本順序は `従来はどうだったか → 何が問題か → 新しい方法はどう解決するか`。**ただしこれは「型」であって「穴」ではない。** 全章を同じ見出しで割らない。
- 機能紹介は `仕組み → 使い方 → 判断基準 → 実務上の注意` の順に置く。
- 単なる API 紹介にしない。背景と設計意図を先に置く。
- 読者がつまずきやすい誤解は、その場で先回りして潰す。
- 必要に応じて旧版や類似概念との差分を短く挟む。
- 結論は急がず、前提を揃えてから示す。

## 4. サンプルコードルール

- コードは説明対象に最短距離で届く量にする。長すぎる完成品は避ける。
- **1 つの章には、原則として段階的に積み上がる複数のコード例を置く。** 自明な 1 スニペットだけで「具体例で見る」を済ませない。
- 1 つの概念につき、最小のコード例をまず出す。
- 必要なら `HTML/JSX/ERB` と `CSS` を対で見せる。
- コードブロックには必ず言語ラベルを付ける。**`txt` でごまかさず、実在する言語で書く。**
- コードの前後で「何を見るべきか」を本文で明示する。
- 生成物を示すときは、簡略化であることを断る。
- 実務章では、その技術らしい自然なコードを使う。Rails 章は Rails の書き味に寄せる。
- 悪い例と良い例の対比を短く出すのは有効。
- コード例は網羅ではなく判断基準の説明に従属させる。

## 5. 図版ルール

- 図は装飾ではなく説明の圧縮手段として使う。
- 1 図 1 主張。図だけで複数論点を背負わせない。
- 抽象概念の関係、流れ、対比、空間の違いを示すときに使う。
- 本文で説明しづらい「歴史の流れ」「役割の違い」「段階的な関係」は積極的に図にする。
- 図の前後で本文が図の読み方を支える。
- キャプションは短く言い切る。本文の繰り返しにしない。
- 色だけに依存しない。ラベルや配置で意味を補う。
- 図中のコード断片は本文と一致させる。

## 6. コラムルール

- この書籍では、独立した「コラム」節はほぼ使わない。
- 脇道情報は `引用ブロック` で短い注意や補足を入れるか、`付録` にまとまった補助情報を逃がすか、`演習` で再構成を促す。
- 本筋を崩さずに補助導線を作る。コラムを使うなら、本筋を遮らない短い補足か判断基準に限る。

## 7. 初学者への配慮

- 完全初心者向けではない前提を明示する。
- ただし初学者がつまずく点は丁寧に救う。
- 用語は初出で短く定義する。
- 「今すぐ試したい人向け」の別導線を用意する。
- 失敗しやすいポイントは症状別に切り分ける。
- 読み方を複数提示する。順読、つまみ読み、先にハンズオン、など。
- 読者に求める前提知識は最初に明言する。
- 読者に自分の言葉で説明させる演習を置く。

## 8. 技術的正確性ルール

- バージョン前提を明記する。
- 過去バージョンの話は必ず条件付きで書く。
- 曖昧な一般論で済ませず、仕組みの根拠を書く。
- 一次情報を章末に置く。
- 実務上の例外や限界も隠さない。
- 読者が古い記事で混乱しそうな点は明示的に差分を書く。
- 便利さより正確さを優先する。

## 9. FBC Press の思想

- 単なるハウツー本ではなく、「背景・思想・仕組み・実務判断」を一続きで扱う。
- 機能の暗記より、道具が生まれた理由を重視する。
- 技術を賛美しすぎない。批判や限界も正面から扱う。
- 実務での判断材料を渡すことが目的。
- 初学者を甘やかしすぎず、しかし突き放さない。
- 一貫性、保守性、再利用性、設計判断を重視する。
- 公式情報と現場感覚の両方をつなぐ。

## 10. 著者の説明スタイル

- 教師的だが説教臭くない。
- 先に論点を立て、次に理由を解く。
- 比較、歴史、対比表が多い。**ただし表を「置くこと」が目的ではない。** その表が議論のどこで必要になったかを本文が支える。
- 「これは何ではないか」も使って輪郭を明確にする。
- 実務でどう痛むかを具体的に書く。
- 読者に判断基準を渡す書き方が中心。
- 章をまたぐ参照が多く、全体を体系として読ませる。

## 11. 用語の導入方法

- 初出で日本語説明を添える。
- カタカナ語は、必要なら英語原語や具体例で固定する。
- 用語は揺らさない(`TERMS.md` を唯一の基準にする)。
- 用語集を付録にまとめ、本文から参照可能にする。
- 重要語は「定義 → 例 → 注意点」の順で入れる。
- 似た概念の混同は、その場で明確に分ける。

## 12. やってはいけないこと

**まず、過去に実際に本をダメにした失敗を名指しで禁止します(第0節参照)。**

- 🚫 **全章で同じ節見出しのセットを使う**(`まず押さえたいこと / 具体例で見る / なぜそうなるのか / まとめ` のような型の使い回し)。
- 🚫 **タイトルを差し込んだだけの定型まとめ**を書く。まとめは毎章、その章固有の内容で書く。
- 🚫 **自明な 1 スニペットだけで「具体例」を済ませる**。例は積み上げる。
- 🚫 **タイトルの語を別物に変えても通る一般論**で章を埋める。その章にしか書けない中身を書く。
- 🚫 **主張の言い換えで段落を水増しする**。新しい段落は新しい情報を運ぶ。

加えて、従来からの禁止事項:

- クラス一覧や API 一覧の羅列だけで章を作る。
- 背景説明なしに結論だけを押しつける。
- 長いコードを丸ごと貼って説明した気になる。
- 初出用語を未定義のまま使う。
- 旧バージョン情報を現行仕様と混ぜる。
- 便利さだけを強調して欠点を隠す。
- コラムで本筋を頻繁に中断する。
- 図を装飾目的で使う。
- 「絶対」「万能」といった強すぎる断定を多用する。
- 章末の参考資料を省く。
- 読者の前提知識を曖昧にしたまま進める。
- 判断基準を示さず、好みの話に還元する。

## 13. 本固有のスタイルは各書籍リポジトリで決める

このガイドは **シリーズ共通の価値観と水準** を定義します。次は各書籍リポジトリ側の `STYLEGUIDE.md` で具体化してください(既刊『Hotwire 教科書』『Tailwind CSS 教科書』のリポジトリが手本です)。

- その本のバージョン前提(対象バージョン、確認日)
- その本のドメイン用語の統一表
- その本で使う言語ラベルの一覧
- 確認済みの基準 URL

**汎用ガイドだけで章を書かせない。** 中身の設計は、各書籍の `OUTLINE.md` に章ごとの論証として書き起こしてから執筆します(`BOOK_OUTLINE_TEMPLATE.md` 参照)。中身が設計されていないと、書き手は骨格を filler で埋めるしかなくなります。これが『HTML 雑学』で起きたことの本質的な原因でした。

## 補足

この既刊の特徴は「コラムを立てない」「章末参考資料を必須化する」「背景と判断基準を最優先する」の 3 点です。  
そして最も大事なのは、**形(見出し・節・参考資料)を揃えることではなく、密度(具体例・対比・根拠・体系的接続)を揃えること** です。形だけ真似ると授業ノートになります。第0節を繰り返し参照してください。
