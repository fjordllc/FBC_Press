# 第5章 tbody は「ブラウザの親切」ではなく表モデルの本体である

前章では、ブラウザが `html`・`head`・`body` を補う様子を見ました。あれは「省略された外枠を補う」話でした。この章で扱う `tbody` は、見かけは似ていても性質が違います。ブラウザは **書かれていない構造を足している** のではなく、**表という仕組みがもともと持っている階層を、ソースの省略にかかわらず必ず立ち上げている** のです。

この違いが分かると、「なぜ自分の CSS や JavaScript が表で空振りするのか」という実務のつまずきが、まとめて説明できるようになります。それがこの章のゴールです。

## 5.1 表は「行の集まり」ではなく「行グループの集まり」

多くの人は、表を「行(`tr`)が縦に並んだもの」だと考えています。しかし HTML の表モデルでは、`table` の直接の子は行ではありません。**行グループ**——`thead`(見出し行のまとまり)、`tbody`(本体行のまとまり)、`tfoot`(脚注行のまとまり)——です。行(`tr`)は、その行グループの中に入ります。

```text
table
├─ thead
│   └─ tr
├─ tbody        ← 本体行は必ずこの中
│   ├─ tr
│   └─ tr
└─ tfoot
    └─ tr
```

つまり表は最初から **2 段階の入れ子** を前提にした構造です。`thead` と `tfoot` を「見出しと脚注のまとまり」として認めるなら、本体行にも対応するまとまりが要る——その本体側のまとまりが `tbody` です。`tbody` だけが特別なのではなく、3 つで 1 組です。

## 5.2 だから、書かなくてもブラウザは tbody を立てる

ここで、本体行だけのいちばん素朴な表を書いてみます。

```html
<table>
  <tr><td>A</td></tr>
</table>
```

ソースに `tbody` はありません。ところが DevTools で DOM を見ると、こうなっています。

```html
<table>
  <tbody>
    <tr><td>A</td></tr>
  </tbody>
</table>
```

`tr` は `table` の直接の子ではなく、**間に挟まった `tbody` の子** になりました。これは「ブラウザが気を利かせて整えた」のではありません。表モデルでは本体行は行グループの中にしか存在できないので、行グループが省略されていれば **必ず立ち上げる** ——仕様で決まった動作です(`thead` を自分で書いたときは、そちらが使われるので本体の `tbody` だけが補われます)。

ここで一度、`View Source`(あなたが書いたソース)と DevTools(ブラウザが構築した DOM)が **別物** だと押さえてください。前者に `tbody` はなく、後者にはある。この章のすべての実務上の落とし穴は、この一点から派生します。

## 5.3 これがなぜ実務で痛むのか — CSS と JavaScript が空振りする

「DOM に `tbody` が増えるだけなら、知識としては面白いが実害はない」と思うかもしれません。ところが、CSS と JavaScript は **DOM を相手にする** ので、ソースに無い `tbody` が両方に効いてきます。

まず CSS。表の直下の行だけを狙ったつもりの、次のセレクタを見てください。

```css
/* 「table の直接の子の tr」を狙ったつもり */
table > tr {
  background: #eee;
}
```

これは **1 行にも当たりません**。DOM 上では `tr` の親は `table` ではなく `tbody` だからです。意図どおりに当てるには、間の行グループを通します。

```css
/* DOM の実態に合わせる */
table > tbody > tr {
  background: #eee;
}
/* あるいは子結合子をやめて、すべての子孫の tr を狙う */
table tr {
  background: #eee;
}
```

次に JavaScript。`table` の子を直接たどると、やはり外します。

```js
const table = document.querySelector('table');

// ソースの見た目から「table の子は tr」と思い込むと空振りする
table.children;            // → [tbody]  (tr ではない!)
table.children[0].tagName; // → "TBODY"

// 行が欲しいなら、表が用意している API を使う
table.rows;                // → [tr, tr, ...]  行グループをまたいで集める
table.tBodies[0].rows;     // → 本体行だけ
```

`table.rows` のような専用 API が用意されているのは偶然ではありません。**行グループをまたいで行を集める必要が頻繁にある** と分かっているから、DOM 側がショートカットを用意しているのです。API の存在自体が、表が階層構造であることの証拠になっています。

## 5.4 「だったら最初から必須にすればよかった」への答え

ここまで読むと、自然な疑問が湧きます。「DOM では必ず `tbody` が要るなら、ソースでも必須にして、補完なんてやめればよかったのではないか」。

これは HTML の設計思想を理解する良い入り口です。HTML は **書き手が厳密でないことを前提に作られた言語** です。手書きの表で毎回 `tbody` を律儀に書く人は多くありません。そこで HTML は、「内部のモデルは厳密に保ちつつ、入力としては省略を受け入れる」という折衷を選びました。XML なら省略はエラーですが、HTML は省略を **回復可能な普通の入力** として扱います。

この「内部は厳密・入力は寛容」という二層構造こそ、`tbody` の補完、`p` の自動終了(次章)、壊れた HTML の回復(第7章)に共通する、HTML 全体を貫く設計です。`tbody` はその最も分かりやすい一例にすぎません。

## 5.5 まとめ

`tbody` の自動挿入は「ブラウザの小さな親切」ではありません。表が最初から **行グループの入れ子** として設計されていて、ブラウザはその構造をソースの省略にかかわらず立ち上げているだけです。

この見方を持つと、バラバラに見えた症状が 1 本につながります。`table > tr` が効かないのも、`table.children` が `tr` を返さないのも、`table.rows` という専用 API があるのも、すべて「本体行は `tbody` の中にいる」という一点から導けます。豆知識として `tbody` を覚えるのと、表モデルとして理解するのとでは、デバッグの速さがここで変わります。

次章では、同じ「内部は厳密・入力は寛容」という設計が、`p` 要素の暗黙の終了タグにどう現れるかを見ます。

## この見本のどこが「FBC らしさ」か(執筆者向けメモ)

- 見出しがすべて **主張**(「tbody は親切ではなく表モデルの本体である」「CSS と JavaScript が空振りする」)。役割名がない。
- コード例が **積み上がる**:素朴な表 → 補完後の DOM → 効かない CSS → 直した CSS → 外す JS → 専用 API。前の例が次の例の前提になっている。
- 読者の **実害(CSS/JS の空振り)** に着地させ、知識を判断に変えている。
- 「だったら必須にすれば?」という **誤解を先回り** して、設計思想に接続している。
- まとめが **この章固有**。タイトルを別の語に変えたら成立しない。
- 他章(前章・次章・第7章)と **体系として接続** している。
- それでも 1 文は短く、コードは最小。**「短い文」と「濃い章」は両立する。**
