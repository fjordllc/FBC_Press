# FBC Press Context Management

この文書は、FBC Press の長期執筆・レビュー作業で、会話履歴だけに重要情報を残さないための運用ルールです。

## 基本方針

長期作業によるコンテキスト消費を抑えるため、作業の節目で `/compact` を実行します。

ただし、単純なメッセージ数や一定時間を基準に機械的に実行してはいけません。現在進行中の判断や未解決事項が失われないよう、作業単位が完了したタイミングで実行します。

## `/compact` を実行する主なタイミング

- `OUTLINE.md` などの設計文書が確定した後
- 一章分の章設計が確定した後
- 初回レビューへの対応と再レビューが完了した後
- 一章を完了し、次章へ進む前
- 長い調査や議論が終わり、結論がリポジトリ内の Markdown へ記録された後
- 残りコンテキストが少なくなり、以後の作業へ影響しそうなとき

## `/compact` の前に行うこと

実行前に、その時点で今後必要になる情報がリポジトリ内へ保存されていることを確認します。

最低限、次を記録します。

- 確定した方針
- 未解決の課題
- 採用しなかった案と理由
- 現在の作業状況
- 次に行う作業
- 関係するファイル
- 実行・検証結果
- レビューへの対応状況

必要に応じて、章ごとの `changes.md`、`review-summary.md`、作業用の `STATUS.md` などを更新します。

## `/compact` を避けるタイミング

- 原稿やコードの変更途中
- レビュー指摘への対応途中
- 技術的な調査結果をまだ文書へ記録していないとき
- 複数案を比較しており、結論が出ていないとき
- テストや動作確認が未完了のとき

## 原則

会話履歴をプロジェクトの唯一の記録場所にしてはいけません。

重要な判断、仕様、レビュー結果、進行状況は必ずリポジトリ内の Markdown へ残し、`/compact` 後も作業を正確に再開できる状態を維持します。
